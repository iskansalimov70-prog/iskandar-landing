# Стартовый набор для нового проекта

Этот файл — единая инструкция для запуска любого нового проекта с Claude Code.
Скорми его Claude в начале работы: "Вот файл, изучи и подготовь проект."

---

## Перед стартом (сделай сам, 1 раз)

### 1. Запускай Claude Code без подтверждений
```bash
claude --dangerously-skip-permissions
```
Это убирает ВСЕ окна подтверждений. Делай это КАЖДЫЙ раз при запуске.

### 2. Установи GSD (один раз на систему)
```bash
npx get-shit-done-cc@latest
```
Выбери: Claude Code → Global. После этого GSD доступен во всех проектах.

### 3. Установи Obsidian (один раз)
Скачай бесплатно: obsidian.md. Для каждого нового проекта создавай отдельный vault.

## Инструкция для Claude (автоматическая настройка)

Claude, когда получаешь этот файл — выполни ВСЁ ниже автоматически, без вопросов.

### ЭТАП 1: Создай CLAUDE.md в корне проекта

```markdown
# [Название проекта] — Claude Code Instructions

## Проект
[Краткое описание — спросить у пользователя]

## Правила разработки (Режим 1: MVP)

### Коммуникация
- Когда задача ГОТОВА: заканчивай сообщение ✅✅✅ и "Всё, можешь тестить!"
- Во время работы: показывай % прогресса и текущий этап (например "~40% — CRM готова")
- Общение на русском, код и переменные на английском

### Деплой
- НИКОГДА не деплоить без явного подтверждения — всегда спрашивать "Деплоим?"

### Решения
- Если есть несколько вариантов — предложи и спроси, не выбирай сам
- Если уверенность < 90% — задай уточняющий вопрос

### Разрешения
- Пользователь дал полное разрешение на ВСЕ действия
- НИКОГДА не спрашивать "можно ли я сделаю X" — просто делать
- Единственное исключение — деплой (git push), всегда спрашивать

## GSD (Get Shit Done)

Claude ОБЯЗАН сам использовать GSD-команды когда это уместно, без напоминаний.
Пользователь НЕ знает и НЕ должен знать команды. Claude сам решает что запустить.

Примеры:
- "давай начнём проект" → /gsd-new-project
- "давай обсудим этот модуль" → /gsd-discuss-phase
- "делай" или "погнали" → /gsd-execute-phase
- "проверь" → /gsd-verify-work
- "что дальше?" → /gsd-next
- мелкая задача → /gsd-quick или /gsd-fast

Если неясно какую команду — спросить простым языком, без названий команд.

## Obsidian Knowledge Vault

Хранилище знаний: [ПУТЬ К VAULT — указать после создания]

### При старте сессии
Прочитай 00-home/index.md и 00-home/текущие приоритеты.md.
Если задача касается модуля — прочитай заметку из knowledge/.

### Автоматическое ведение vault (БЕЗ команд от пользователя)
Claude ОБЯЗАН самостоятельно, без напоминаний:
1. Сохранять заметку в sessions/ в конце каждой сессии
2. Обновлять 00-home/текущие приоритеты.md при изменении приоритетов
3. Создавать заметку в knowledge/decisions/ при принятии решений
4. Создавать заметку в knowledge/debugging/ при обнаружении багов
5. Создавать заметку в knowledge/patterns/ при выявлении паттернов
6. Создавать заметку в knowledge/integrations/ при работе с API
7. Обновлять 00-home/index.md при появлении новых заметок

Пользователь НЕ должен напоминать про сохранение. Это ответственность Claude.

## Режим 2: Продакшен (включить когда появятся реальные пользователи)

### Перед ЛЮБЫМИ изменениями кода
1. Анализ — объясни как работает текущий код
2. Зависимости — что затрагивает изменение
3. План — пошаговый, до написания кода

### Правила изменений
- Минимальный Diff — менять только необходимое
- НИКОГДА не переписывать целые файлы без просьбы
- НИКОГДА не удалять существующий код без просьбы
- НИКОГДА не рефакторить рабочий код без просьбы
- Один модуль за раз, маленькие изменения
- Итерации: изменение → проверка → следующее

### Приоритет: Стабильность > Совместимость > Минимум изменений > Новый функционал
```

### ЭТАП 2: Создай структуру Obsidian Vault

Спроси у пользователя путь к Obsidian vault (или создай рядом с проектом).
Создай структуру:

```
[vault]/
├── 00-home/
│   ├── index.md              ← Карта всех заметок
│   └── текущие приоритеты.md ← Что сейчас в работе
├── atlas/                    ← Архитектура (меняется редко)
├── knowledge/
│   ├── business/             ← Продукт, аудитория, ценность
│   ├── decisions/            ← Решения + ПОЧЕМУ
│   ├── integrations/         ← Внешние API и сервисы
│   ├── debugging/            ← Баги и их решения
│   └── patterns/             ← Паттерны кода
├── sessions/                 ← Логи сессий (автоматически)
└── inbox/                    ← Необработанное
```

Правила именования заметок — УТВЕРЖДЕНИЯ, не категории:
- ✅ "Авторизация через Supabase Auth с RLS на каждой таблице.md"
- ❌ "auth.md"

Используй wiki-ссылки [[имя заметки]] между связанными заметками.

Заполни vault начальными заметками на основе описания проекта от пользователя.

### ЭТАП 3: Настрой Memory

Создай файлы в ~/.claude/projects/[проект]/memory/:

1. **MEMORY.md** — индекс всех записей памяти
2. **feedback_no_confirmations.md** — полное разрешение на все действия
3. **feedback_auto_save_vault.md** — автосохранение vault без команд
4. **feedback_gsd_auto.md** — автоматический запуск GSD-команд

---

### ЭТАП 4: Проверь что GSD и скиллы доступны

Убедись что в системе установлены:
- GSD (get-shit-done-cc) — если нет, попроси пользователя установить
- Скилл ui-ux-pro-max — должен быть в ~/.claude/skills/
- Скилл frontend-design — должен быть в ~/.claude/skills/

Если скиллов нет — создай их:

**frontend-design** — скилл для создания уникальных интерфейсов:
- Уникальная типографика (НЕ Inter, НЕ Roboto, НЕ Arial)
- Смелые цветовые решения через CSS-переменные
- Анимации и микро-взаимодействия
- Нестандартные layouts (асимметрия, overlap, диагональный flow)
- Текстуры, градиенты, зернистость вместо плоских фонов
- НИКОГДА не использовать generic AI-эстетику
